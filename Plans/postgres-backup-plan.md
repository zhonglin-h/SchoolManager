# Plan: PostgreSQL Migration + Nightly Google Drive Backup

## Goal
Replace H2 with PostgreSQL for a more robust local database, and add a nightly backup
job that dumps the DB, uploads the compressed file to Google Drive, and prunes old backups.

---

## Phase 1 — Switch to PostgreSQL

### 1.1 `build.gradle`
- Remove `runtimeOnly 'com.h2database:h2'`
- Add `runtimeOnly 'org.postgresql:postgresql'`
- Add `implementation 'org.flywaydb:flyway-database-postgresql'` (Flyway PostgreSQL support)
- Add `implementation 'com.google.apis:google-api-services-drive:v3-rev20240521-2.0.0'`

### 1.2 `application.properties`
Replace H2 datasource block:
```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/schooldb
spring.datasource.driver-class-name=org.postgresql.Driver
spring.datasource.username=school
spring.datasource.password=<password>
spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect
spring.h2.console.enabled=false   # remove entirely
```

Add backup properties:
```properties
# Nightly backup
app.backup.enabled=true
app.backup.cron=0 0 2 * * *
app.backup.drive-folder-id=<Google Drive folder ID>
app.backup.retention-days=30
app.backup.postgres.host=localhost
app.backup.postgres.port=5432
app.backup.postgres.database=schooldb
app.backup.postgres.username=school
app.backup.postgres.password=<password>
app.backup.pg-dump-path=pg_dump
```

### 1.3 Flyway migrations
All three existing migrations (V1–V3) use standard ANSI SQL and require **no changes** —
`BIGINT GENERATED BY DEFAULT AS IDENTITY`, window functions, and DDL all work on PostgreSQL.

### 1.4 One-time setup (manual, outside the app)
```sql
-- Run once as a Postgres superuser
CREATE USER school WITH PASSWORD '<password>';
CREATE DATABASE schooldb OWNER school;
```

## Phase 2 — Backup Service

### 2.1 `BackupProperties.java` (`config/`)
`@ConfigurationProperties(prefix = "app.backup")` bean holding all `app.backup.*` fields:
`enabled`, `cron`, `driveFolderId`, `retentionDays`, `pgDumpPath`, and a nested
`Postgres` record (`host`, `port`, `database`, `username`, `password`).

### 2.2 `DriveBackupService.java` (`service/`)
Spring `@Service` with a `@Scheduled(cron = ...)` method (cron sourced from properties).
Runs only when `app.backup.enabled=true`.

**`runBackup()` steps:**
1. Build filename: `schooldb-<yyyy-MM-dd-HHmm>.sql.gz`
2. Invoke `pg_dump` via `ProcessBuilder`, pipe stdout through `GZIPOutputStream` to a temp
   file in `./backups/`. Set `PGPASSWORD` env var so no password prompt.
3. Upload the temp file to Google Drive folder (`driveFolderId`) using the Drive API
   (`Files.create` with `application/gzip` MIME type).
4. List all files in the folder, filter by name prefix `schooldb-`, sort by `createdTime` 
   descending, delete any beyond `retentionDays` (compare `createdTime` to `LocalDate.now()`).
5. Delete the local temp file.
6. Log outcome (success or failure with exception message).

### 2.3 `GoogleConfig.java` — add Drive scope + bean
Add `DriveScopes.DRIVE_FILE` to the `SCOPES` list so the OAuth consent includes Drive.  
Add a `Drive` bean (analogous to the existing `Calendar` bean):
```java
@Bean
public Drive googleDrive(NetHttpTransport httpTransport, Credential credential) {
    return new Drive.Builder(httpTransport, GsonFactory.getDefaultInstance(), credential)
            .setApplicationName("SchoolManager")
            .build();
}
```
Note: adding a new scope will require the user to re-authorise once (the stored token will be
automatically refreshed by the existing `FileDataStoreFactory` flow).

## Phase 3 — Restore Procedure (document in README)

```bash
# 1. Download the backup file from Google Drive, e.g. schooldb-2025-06-01-0200.sql.gz

# 2. Ensure the target database exists and is empty (or drop & recreate)
psql -U postgres -c "DROP DATABASE IF EXISTS schooldb;"
psql -U postgres -c "CREATE DATABASE schooldb OWNER school;"

# 3. Restore
gunzip -c schooldb-2025-06-01-0200.sql.gz | psql -U school -d schooldb

# 4. Start the app — Flyway will detect that migrations are already applied and skip them.
cd backend && ./gradlew bootRun
```

## Phase 4 — Update `setup.ps1` and `README.md`

### 4.1 `setup.ps1` — add PostgreSQL check (renumber to [1/7]…[7/7])

Insert a new step 2 between the Java check and the Node.js check:

```powershell
# ---------------------------------------------------------------------------
# 2. Check PostgreSQL
# ---------------------------------------------------------------------------
Write-Host "[2/7] Checking PostgreSQL..."
$pgFound = $true
try {
    $pgVersion = (& psql --version 2>&1).ToString().Trim()
} catch {
    $pgFound = $false
}
if (-not $pgFound -or $LASTEXITCODE -ne 0) {
    Write-Host " ERROR: PostgreSQL (psql / pg_dump) is not installed or not on PATH."
    Write-Host " Install PostgreSQL 16 with:"
    Write-Host "   winget install PostgreSQL.PostgreSQL"
    Write-Host " Then re-run setup.ps1."
    throw "PostgreSQL not found"
}
Write-Host " OK - $pgVersion"
```

Renumber all subsequent steps: Node.js → [3/7], pnpm → [4/7], frontend deps → [5/7],
backend deps → [6/7], credential config → [7/7].

> Add these scopes:

**New "Backups & Restore" section** (at the end, before or after Daily Use):

> ### Backups
> The app runs a nightly `pg_dump` at 2 AM, compresses it, uploads it to the configured
> Google Drive folder, and deletes copies older than 30 days. No manual action required.
>
> ### Restoring from a backup
> 1. Download the `.sql.gz` file from Google Drive.
> 2. Recreate the database:
>    ```powershell
>    psql -U postgres -c "DROP DATABASE IF EXISTS schooldb;"
>    psql -U postgres -c "CREATE DATABASE schooldb OWNER school;"
>    ```
> 3. Restore:
>    ```powershell
>    cmd /c "gunzip -c schooldb-2025-06-01-0200.sql.gz | psql -U school -d schooldb"
>    ```
> 4. Start the app normally — Flyway detects that migrations are already applied and skips them.

---

## Out of scope
- Remote Postgres (all DB traffic stays on localhost)
- Point-in-time recovery / WAL archiving (daily snapshots are sufficient)
- Encrypting the backup file (Drive access control is enough for this use case)
